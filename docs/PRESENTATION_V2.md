# 项目进展汇报
## 迭代四+五：根因诊断、对照实验与覆盖扩展

---

## 我们做到哪了

课题：从 OpenLibrary Python 代码仓库提取概念、构建知识图谱、辅助 issue 定位

| 阶段      | 主要工作                                         |
| --------- | ------------------------------------------------ |
| 迭代一    | 概念过滤重写、LLM 定义修复、下游定位模块从零搭建 |
| 迭代二    | 同义词双轨检测、一词多义聚类、共现子域分桶       |
| 迭代三    | 语义导航层：embedding 检索 + 3-hop BFS 图游走    |
| 迭代四+五 | 根因分析 + 对照实验 + 全量覆盖扩展               |

**当前结果**：

| 指标               | KG-walk   | BM25（基线） |
| ------------------ | --------- | ------------ |
| File Recall@10     | **84.6%** | 92.3%        |
| File MRR           | 0.552     | 0.758        |
| Function Recall@10 | 74.7%     | 84.6%        |

---

## 迭代四要解决什么问题

迭代三结束时，KG 的 File Recall@10 = 82.4%，比 BM25 低约 10 个百分点。

这 10 个百分点差在哪里？还能不能追上去？要回答这个问题，不能靠猜——需要**从数据里找出每一条失败 issue 失败的原因**，然后针对性地验证。

所以迭代四+五做了两件事：
1. 逐条分析失败 case，定位根因
2. 针对每个可能的根因，做对照实验看能不能涨

---

## 第一步：搞清楚谁在失败、为什么失败

写了一个分析脚本，对 91 条 issue 逐一检查：
- issue 的关键词在 KG 的 3,760 个概念里存在吗？
- 匹配到的概念在图中有边吗？还是孤立节点？
- 目标文件（GT file）本身在 KG 的概念索引里吗？

**先分类**：

| 分类                 | 数量   | 含义              |
| -------------------- | ------ | ----------------- |
| 两个方法都命中       | 74     | KG 和 BM25 都正确 |
| KG 命中、BM25 未命中 | **1**  | KG 独占优势       |
| BM25 命中、KG 未命中 | **10** | KG 可以改进的空间 |
| 两个方法都未命中     | 6      | 硬 case           |

这 10 条 KG 失败但 BM25 成功的 case，是这次要解决的目标。

---

## 第二步：逐条逆向追溯

对每条失败 issue，检查 GT 文件在 KG 中的概念覆盖、连通状态、与 issue 关键词的重叠：

**发现一：5 条 case —— 目标文件根本没进 KG**

这些文件位于 `scripts/` 目录下（如 `scripts/providers/isbndb.py`、`scripts/monitoring/monitor.py`）。KG 的 include pattern 只索引了 `openlibrary/**/*.py`，scripts/ 下的 61 个 Python 文件完全不在覆盖范围。BM25 能找到它们（排名第 1），因为 BM25 索引了全部文件的全文。

**发现二：3 条 case —— issue 文本太短**

标题仅 "### Title"，没有任何关键词能匹配到 KG 概念集。

**发现三：2 条 case —— 排名阶段被大文件挤出**

GT 文件的已连接概念数远少于竞争文件，在累加聚合下被挤出前 10。

---

## 第三步：动手前先量化组件贡献

在改任何东西之前，先搞清楚**现有管道里哪个组件贡献最大**。如果改错了地方，可能得不偿失。

**问题**：embedding 语义检索到底有多重要？关掉它会怎样？

**方法**：只改变入口方式（embedding vs 纯 token match），其余全部不变。

| 入口方式                 | File Recall@10 |
| ------------------------ | -------------- |
| 纯 token match           | 63.7%          |
| + embedding 语义检索     | 82.4%          |
| **embedding 的独立贡献** | **+18.7pp**    |

embedding 解决了"issue 用自然语言、KG 用标识符"之间的语义鸿沟——没有它，超过 1/3 的 issue 在入口就断了。这是整个管道最强的单一组件。

---

## 迭代五：尝试突破 82.4%

根因分析指出了三个可能的改进方向。针对每个方向做对照实验——改一个变量，看结果变化。

### 尝试一：混合评分

**思路**：KG 和 BM25 可能捕捉的是不同类型的信号。用 rank-level 融合（Reciprocal Rank Fusion）合并两个排名，看能否互补。

按不同权重比扫描：

| BM25 权重       | Recall@10 | 说明        |
| --------------- | --------- | ----------- |
| 0%（纯 KG）     | 83.5%     | —           |
| 50%             | 90.1%     | —           |
| **60%**         | **92.3%** | 追平纯 BM25 |
| 100%（纯 BM25） | 92.3%     | —           |

**结果**：混合可以追平 BM25，但无法超越。因为 RRF 是"共识融合"——两个方法都认可的文件得分高。KG 独占正确的那条 issue 在 BM25 侧得零分，被淹没了。要让混合超越 BM25，需要 KG 先独立变强。

---

### 尝试二：修复排名公式

**思路**：溯源发现大文件（如 `core/models.py` 有 179 个概念）在累加聚合下"积少成多"，即使与 issue 不直接相关也能排进前 10。如果能消除这个偏向，小但相关的文件可能获益。

尝试了三种修复，每次只改一个变量：

| 尝试 | 方法                        | Recall@10 | 变化    |
| ---- | --------------------------- | --------- | ------- |
| 原始 | per-occurrence SUM          | 82.4%     | —       |
| A    | 除以 sqrt(文件概念数)       | 65.9%     | -16.5pp |
| B    | 每对 (概念, 文件) 只计 1 次 | 70.3%     | -12.1pp |
| C    | 种子概念 3× 权重            | 60.4%     | -22.0pp |

三次全部倒退。不是说大文件偏见不存在——而是当前公式在 74 条成功 case 上已经调得很精细，任何改动都会破坏更多成功 case。

**结论**：排名公式不是瓶颈。这条路走不通。

---

### 尝试三：Call Graph 补边

**思路**：KG 中超过一半的概念是孤立节点（没有任何边）。如果用 Python 函数调用图给孤立概念建立新边，BFS 就能走通更多路径。

从 353 个 .py 文件提取了 3,288 个函数的 11,758 对调用关系。尝试了不同力度的边注入：

| 力度                     | 边数  | Recall@10  |
| ------------------------ | ----- | ---------- |
| 大量（所有调用对）       | 162k  | BFS 被淹没 |
| 中量（一侧孤立）         | 80k   | 63.7%      |
| 少量（严格控制，低权重） | 12.5k | 63.7%      |

**困境**：边多了 → 噪声淹没信号。边少了 → 权重太低（0.08 × 0.2 = 0.016），穿透不了 3-hop BFS 的三层衰减。

**结论**：代码调用 ≠ 概念语义相关。这条路也走不通。

---

### 尝试四：扩展文件覆盖，全量重跑管道

**思路**：5 条失败 case 是因为 scripts/ 目录没被索引。把 include pattern 扩展，然后**全量重跑 Phase 1-6**，让新概念参与同义词检测和共现检测，与旧概念建立边。

之前做过一次添加 scripts/ 概念但不重跑关系检测的实验——新概念 100% 成为孤立节点，结果没变。这次的关键是**跑完整管道**。

| 指标               | 重跑前    | 重跑后    |
| ------------------ | --------- | --------- |
| 索引文件           | 223       | **285**   |
| 概念数             | 3,760     | **4,338** |
| 同义词对           | 1,382     | **1,530** |
| 总边数             | 2,562     | **2,825** |
| **File Recall@10** | **82.4%** | **83.5%** |
| File MRR           | 0.547     | **0.557** |
| Function Recall@10 | 73.6%     | **76.9%** |

**+1.1 个百分点。** 涨了，但增量远小于"5 条全修"的预期。新概念的边密度仍然远低于核心概念，BFS 能传导的信号有限。不过这次重跑为后续的 KG 质量修复奠定了基础——概念池扩大了，后续的关系优化才能生效。

---

### 尝试五：LLM 翻译 issue 文本

**思路**：3 条失败 case 的 issue 关键词不在 KG 里。LLM 可以把"用户报 bug 的自然语言"翻译成 KG 里的概念术语。

对 91 条 issue 逐条调用 LLM。翻译质量很高——例如 `/lists/add returns 500 error` 被翻译为 `makelist, setvalue, from_input`（全部是该 issue 的 GT 函数名）。

但翻译出来的术语只有 34.9% 能匹配到 KG 概念集。两种注入方式（追加到文本、过滤后注入）均未改变 Recall@10。

**结论**：瓶颈不在于"issue 说的词 KG 里没有"——而在于很多目标文件本来就不在 KG 覆盖里，入口再准也没用。

---

## 回顾：KG 本身有没有质量问题？

上述五个实验中，四个没有效果。这引出一个根本问题：**我们构建的知识图谱本身是否存在质量问题，导致下游定位被限制？**

对此我们做了三方面的诊断。

### 诊断一：一词多义——DBSCAN eps 过紧

| 指标          | eps=0.35（旧） | eps=0.55（新）    |
| ------------- | -------------- | ----------------- |
| `date` 聚类数 | **69**         | 11                |
| `book` 聚类数 | **68**         | 9                 |
| `user` 聚类数 | **51**         | 8                 |
| `work` 聚类数 | **45**         | 4                 |
| 总簇数        | 7,533          | **2,826（-62%）** |

旧 eps=0.35 把 LLM 定义措辞的细微差异当成"不同含义"切开。例如 `date` 的 "A date is a specific calendar day used to identify..." 和 "A date is a calendar day used to track..." 被切成两个不同的簇。多义消歧模块锁定一个簇后，丢弃了其余 98.5% 的 occurrence——等价于随机阉割了 `date` 概念。

**修复结果**：eps 调到 0.55，File Recall@10 从 83.5% → **84.6%（+1.1pp）**。

---


### 诊断二：共现边密度

尝试降低 min_count（3→2）以增加边数。边数从 2,825 增至 5,478，但 Recall 反而降到 83.5%。边翻倍带来了噪声，冲淡了原本精准的共现信号。**当前阈值已是经验最优。**

### 诊断三：图拓扑结构

将 KG 按连通分量拆解：

| 结构         | 数量         | 占比      |
| ------------ | ------------ | --------- |
| 巨型连通分量 | 1,133 个概念 | 26.1%     |
| 小型连通分量 | 509 个概念   | 11.7%     |
| 孤立节点     | 2,385 个概念 | **55.0%** |

BFS 从一个种子概念出发，最多只能走到它所在连通分量的边界。对于 55% 的孤立节点，一步都走不出去。

**各分量概念的平均出现频次**：巨型分量 15.2 次 vs 小型分量 2.9 次。小型分量里的概念本身就不常用，即使有边也难以为 issue 定位提供足够强的信号。

---

### 诊断四：15 条失败 issue 的当前瓶颈

| 失败模式                | 数量  | 说明                                         |
| ----------------------- | ----- | -------------------------------------------- |
| 入口缺失                | 5     | issue 标题仅 "### Title"，没有可匹配的关键词 |
| 文件未覆盖              | 2     | GT 文件完全不在 KG 索引范围                  |
| **同组件但未进 top-10** | **7** | BFS 走到了正确区域，但排名没推上去           |
| 不同组件隔离            | **0** | eps 修复后此类已清零                         |

### 诊断结论

- **一词多义过切是真实的 KG 质量问题**，修复后涨了 +1.1pp——验证了你的猜测
- **共现边在当前设计下已达到最优**，降低阈值只增加噪声
- **55% 的孤立节点和碎片化的图拓扑**是当前 KG 最大的结构弱点，但在现有方法框架内（基于 scope 共现 + embedding 同义），没有简单修复方案
- 7 条"同组件但未进 top-10"的失败 case，说明排名阶段的信号竞争是最终瓶颈的末段

## 项目整体回顾：从"建图"到"导航"

这个项目经历了两个大的阶段：

### 第一阶段：建立知识图谱

我们把 OpenLibrary 的 Python 源码变成了一张**可查询的知识图谱**。

| 建设内容     | 结果                                                            |
| ------------ | --------------------------------------------------------------- |
| 概念提取     | 从 285 个文件中提取 24,558 个代码标识符                         |
| LLM 语义定义 | **100% 成功率**，每个标识符都有了一句话的语义定义               |
| 同义词关系   | 1,530 对（包含命名变体和深层语义等价）                          |
| 一词多义检测 | 检测 628 个概念，经 eps 修复后产生 2,826 个语义簇（此前 7,533） |
| 共现关系     | 1,295 对（经子域分桶过滤噪声）                                  |
| **最终 KG**  | **4,338 个领域概念，2,825 条边**，可导入 Neo4j 交互查询         |

---

### 第二阶段：实现导航定位

在图的基础上，构建了**语义导航层**——给定一条 issue，自动找到应该修改的代码文件。

```
Issue 文本 → Embedding 语义检索 → 一词多义消歧 →
3-hop BFS 图游走 → 文件加权排名 → 输出 + 路径解释
```

---

## 核心发现

### 一、实验证据：定位精度的演进

| 阶段                               | File Recall@10 | 关键改进                                             |
| ---------------------------------- | -------------- | ---------------------------------------------------- |
| 首次评估（naive token match）      | ≈ 0%           | Issue 的自然语言和 KG 标识符之间存在巨大语义鸿沟     |
| 引入 embedding 语义检索            | 63.7%          | Embedding 独自贡献 **+18.7pp**，是系统最强的单一组件 |
| 加入多义消歧 + BFS + IDF 排名      | 82.4%          | 迭代三完成时的基线                                   |
| 全量文件覆盖 + 关系重构建          | 83.5%          | 扩展 scripts/ 等缺失目录，重跑完整管道               |
| **一词多义 eps 修复（0.35→0.55）** | **84.6%**      | 聚类从 7,533 降至 2,826，消除过度分簇对消歧的干扰    |

从 82.4% 出发做了 7 组对照实验：排名公式 ×3（全部倒退）、call graph ×3（信号/噪声无解）、全量覆盖（+1.1pp）、eps 修复（+1.1pp）、共现放宽（倒退）、LLM 翻译（无变化）。KG + BM25 混合评分（RRF）追平了 BM25 的 92.3%。

---

### 二、核心认知：KG 和 BM25 是互补的，不是替代关系

KG 用 4,338 个过滤后的领域标识符工作，BM25 用 286 个文件的全文 token。**信号量级的差距**不是"改一个组件"能弥补的——这是两个方法根本性的方法论差异。

**但 KG 在以下维度提供了 BM25 做不到的价值：**

- **语义理解**：当 issue 说的是"借阅限额错误地阻止了零活跃借阅的读者"，这里面没有 `loan.py`、`patron` 这些词——只有 embedding 语义检索能桥接这个距离
- **一词多义消歧**：`user` 的"账户用户"和"数据库角色"两种含义，KG 能锁定正确的簇，丢弃无关路径。BM25 不理解这种语义分层
- **可解释性**：输出不仅告诉开发者改哪个文件，还解释"为什么"——从 issue 关键词到概念到边的完整推理链
- **零 API 消耗**：定位阶段全本地运行，秒级完成，极具工业落地潜力

---

### 三、方法边界：我们精确标定了 KG 在 issue 定位上的适用范围

**当前 KG 的结构限制**：55% 的概念是孤立节点，巨型连通分量仅占 26.1%。BFS 从种子出发后能走到的范围受限。

**15 条失败 issue 的瓶颈分布**：5 条无入口（issue 文本太短）+ 2 条文件未覆盖 + 7 条同组件但排名未进 top-10 + **0 条因组件隔离失败**（eps 修复后此类清零）。

这个项目不是一个单纯的"搜索引擎"——**它的价值在于把这套方法验证到了可交付的程度，并精确标定了方法的边界。** 后续工作可以在此基础上继续推进。

---

## 这个项目最终可以走多远

**我们构建的不只是一个 issue 定位工具，而是一套代码知识基础设施。**

### 近期可达的结果

- **完整的实验报告**：从概念抽取到 issue 定位的完整链路、全部实验数据、消融分析
- **可复现的管道**：一键 pipeline 脚本，任何人可从零跑出相同结果
- **可查询的 KG**：Neo4j 中的概念关系图，支持交互式探索

### 长远价值


- **AI Agent 的"导航地图"**：未来的 AI 编程助手（如 SWE-agent、Devin）可以通过 KG 理解代码库的领域语义，从"定位文件"走向"全自动修复"
- **代码审查辅助**：PR 改动了 `core/loan.py`，KG 可以自动提醒"这个改动可能影响 `waitinglist` 和 `patron` 相关逻辑"
- **新人 onboarding**：新加入项目的开发者可以通过查询 KG（"loan 这个概念在哪里出现？和哪些概念有关？"）快速理解代码库结构
